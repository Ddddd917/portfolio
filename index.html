Project 01 · Architecture

ORION · 个人智能系统 设计阶段

本地优先的个人 AI 系统——不是一个更好的聊天机器人，而是一个真正了解我的智能基础设施。

ORION——名字来自猎户星座，寓意方向感与系统性。展开为 Orchestrated Reasoning and Intelligence Operating Network，定义了系统的核心特征：编排式推理、智能化运作、网络化协同。

设计 ORION 有两个并行的动机。第一是能力个性化：OpenClaw 证明了 AI Agent 能做事，但它是通用的——每次对话都要重新解释背景，不了解我的项目、我的文件、我的偏好。我想要的是一个了解我全部上下文的个人 AI 系统，能做出有针对性的判断而不是通用的回答。第二是数字资产管理：文件分散在多个设备和平台，命名不规范，找不到的情况时有发生。按传统方式手动整理需要耗费大量时间，但 AI 让系统性地解决这个问题变得可能了。

一句话概括 ORION 的目标：让 AI 从通用工具变成个人基础设施。

Project 01 · Data Flow

核心数据流与设计决策

Task Flow
任务流 — 多轮确认、异步执行
「帮我准备和 XX 的见面材料」
语音输入 → [交互层] Whisper STT → [理解层] 意图识别 → 「任务类」 → [理解层] 多轮对话确认细节 → [执行层] Dify Workflow： ├─ 检索 ChromaDB（相关文档） ├─ 查询 SQLite（人物信息） └─ LLM 生成 → 格式化 PDF → [展示层] iPad 实时进度 + 飞书推送

除任务流外，系统还支持查询流（单轮快速问答，走本地 8B 模型，亚秒级响应）和录入流（语音或文字录入生活记录，自动提取实体后写入知识库）。三条链路覆盖日常使用的全部交互模式。

核心产品决策

为什么用三种数据库？
不同数据有不同的最佳检索方式。「阿维塔项目准确率」→ 语义检索（ChromaDB）；「评分最高的电影」→ 结构化查询（SQLite）；「认识的人里谁做 AI」→ 图查询（Neo4j）。多模态存储直接影响回答的准确性和覆盖面。
为什么需要 LLM Router？
「今天天气」用 8B 模型 0.5s 出结果；「分析面试策略」需要 32B 甚至云端 Claude。路由策略决定用户感知的响应速度和回答质量——AI 产品中「成本-质量-延迟」三角的核心权衡。
为什么以语音作为主交互模式？
ORION 的核心使用场景不是坐在电脑前打字，而是在日常生活中随时触发——做饭时问一个问题、走路时录入一条想法、出门前确认日程。语音是这些场景下摩擦最低的交互方式。因此系统的输入输出都围绕语音设计：Whisper 做语音转文字，Edge TTS 做语音合成，DJI Mic Mini 保证拾音质量（STT 的准确率高度依赖音频信噪比）。目标是让交互体验接近「和一个了解你的人对话」，而不是「打开一个 App 输入指令」。

Project 02

文档智能入库助手 已上线

上传文档，AI 自动生成摘要、识别知识域、提取标签、建议标准化文件名，一键归档到本地知识库。

Python Streamlit 智谱 GLM API Claude Code 零手写代码

GitHub · github.com/Ddddd917/doc-ingestion-tool

在搭建 ORION 个人知识库的过程中，文档入库是最大的摩擦点：手动给文件起名、分类到文件夹、提取关键信息——每个文档要花 5-10 分钟，导致知识库建设一直停留在「想做但没做」的状态。这个工具的目标是把入库摩擦降到接近零。

核心产品决策

为什么从单页重构为多页应用？
Phase 2 测试后发现，文档分析只是入口，入库后的管理才是核心使用场景。从单页改为 10 个页面（Dashboard + 上传 + 8 个知识域），涉及 4 个新核心模块的架构重构。
为什么做本地文件直接操作？
这个工具不只是分析文档，而是直接对本地文件执行重命名、移动、归档——比在 Finder 里手动操作更直观，也让 ORION 知识库的文件夹结构始终保持清晰。但文件操作有不可逆风险，所以所有操作都需要用户确认后才执行：AI 出建议，人来校准。删除操作额外加了 checkbox 安全开关防误删。
为什么设计 8 个知识域而不是更多或更少？
太多导致分类模糊，太少导致某些域文档过多。8 个覆盖了日常文档的主要类别，每个域附带 0-1 的置信度分数，支持主域 / 关联域分离。
Dashboard — 标签云、最近入库文档与 AI 自动洞察
AI 分析结果 — 摘要、知识域识别（置信度 95%）、标签提取与入库操作
知识域页面 — 按领域浏览已入库文档，查看摘要与标签
Project 03

Resume Copilot 已上线

暑期求职批量投递后，JD 匹配判断和投递进度管理成了效率瓶颈。这个工具让 AI 自动完成简历匹配分析，并提供投递进度管理看板。

Python Streamlit 智谱 GLM API Claude Code

GitHub · github.com/Ddddd917/personal_resume_analyzer

暑期求职阶段批量投递后，两个问题浮出水面：一是逐条阅读 JD 判断匹配度效率太低，光是「这个岗位值不值得投」就要花十几分钟；二是投递记录分散，无法追踪进度和复盘结果。这个工具的目标是把「读 JD → 判断匹配 → 投递管理」这条链路自动化。

上传简历后自动构建个人画像（数据结构对齐飞书 HR 系统），粘贴岗位 JD 后 AI 完成三维匹配分析（硬性条件 40%、经验匹配 35%、软性匹配 25%），输出匹配度评分、逐条差距诊断和定制化简历优化建议。分析结果自动保存到投递看板，实现从 JD 解析到匹配分析到投递决策的完整工作流。

核心产品决策

为什么个人画像数据结构要对齐飞书 HR 系统？
观察到目标公司（字节、智谱、小米等）大多使用飞书 HR 系统管理招聘流程。与其自己设计一套画像结构，不如直接对齐飞书 HR 的字段标准（基础信息、教育经历 × 多段、实习经历 × 多段、项目经历 × 多段）。这样 AI 做匹配分析时，对照的维度和 HR 实际筛选简历时看的维度是一致的。这不是技术决策，是产品决策——从用户实际场景出发。
为什么用三维评分而不是单一匹配分数？
单一分数掩盖了短板在哪里。三维拆分为硬性条件（40%）、经验匹配（35%）、软性匹配（25%）——硬性条件是门槛，不达标就不用投，所以权重最高；软性匹配最难量化，所以权重最低。面试官看重的不是总分，而是差距诊断：到底差在学历、经验、还是软性能力上。
为什么要做投递看板而不是只做分析？
分析是单次行为，但求职是持续过程。批量投递后，需要追踪每个岗位的状态（未投递 / 已投递 / 已回复 / 面试中 / 已通过 / 已拒绝），按匹配度排序决定优先级，复盘哪些类型的岗位转化率更高。看板把单次分析串成了可管理的工作流。
匹配分析 — 三维评分（硬性 40% / 经验 35% / 软性 25%）与差距诊断
投递看板 — 按匹配度排序、状态追踪、详细分析可展开
开发教训
开发中遇到 API 静默失败——模型名拼写错误（产品名 glm-4.7-flashx vs API 参数 glm-4-flashx），但 API 无报错提示，页面一直转圈。排查走了很多弯路。产品启示：AI 产品的失败必须可见，静默失败是最差的用户体验。

Vibe Coding 与 AI 产品思考

四个项目的实践 + Anthropic Academy 的系统学习，沉淀出的方法论和认知。

Vibe Coding 方法论
Vibe Coding 不是「让 AI 写代码」，而是「用产品思维驱动 AI 开发」。四个项目做下来，我的核心价值归结为三件事：第一，定义做什么和不做什么——比如入库助手决定跳过导出功能优先做文件管理，Resume Copilot 决定画像结构对齐飞书 HR 而非自己设计；第二，做架构决策——比如元数据与文件分离存储、知识库路径放在用户文档目录而非项目目录、三维评分而非单一匹配分数；第三，测试归因——发现命名缺少文档类型维度、API 静默失败的根因是模型名拼写错误，都是我在测试中识别问题后定位原因并给 AI 新 Prompt 修复的。

学习 AI 产品的三个认知

从简单方案开始，只在证明必要时增加复杂度
Anthropic 在《Building Effective Agents》中的核心建议是 "find the simplest solution possible"。这个原则贯穿了我所有项目的技术选型：入库助手用 Streamlit 而不是 React，用本地 JSON 而不是数据库；ORION 的架构设计是混合式的——日常查询走轻量 Routing 快速响应，结构化任务走 Workflow 保证稳定性，只有真正需要动态决策的复杂任务才启动 Agent 模式。不是越复杂越好，而是刚好够用最好。
AI 产品的核心设计挑战是非确定性
传统软件同样的输入一定产出同样的结果，但 Agent 系统中 LLM 的每一步决策都是概率性的——同样的输入可能走不同的路径。这意味着 Bug 难复现、成本难预测、用户无法预期系统行为。我在 Resume Copilot 开发中遇到的 API 静默失败就是一个缩影：模型名拼写错误不会报错，只会静默挂起，排查走了很多弯路。产品设计上，这要求两件事——透明度（让用户看到 AI 在做什么）和可控性（关键操作必须人工确认）。入库助手的「AI 出建议，人来校准」就是这个原则的体现。
AI 产品经理的核心能力是判断力，不是编码能力
Vibe Coding 证明了一件事：代码可以由 AI 生成，但「做什么、不做什么、做到什么程度」这些判断无法自动化。哪个功能该砍（删除面试追踪模块）、数据结构该对齐谁（飞书 HR 系统）、架构该怎么拆（单页重构为多页）、在 Agent 光谱上该站在哪个位置（ORION 的混合架构）——这些产品决策是 PM 不可替代的价值。技术在加速变化，但判断力的积累没有捷径。
